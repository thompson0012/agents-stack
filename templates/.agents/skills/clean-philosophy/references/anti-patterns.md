# Anti-Patterns

Patterns AI must actively avoid. Each entry includes cross-level examples
showing how the same anti-pattern manifests at different Clean levels.

| Anti-pattern | What it looks like | Correct behavior | Why this hurts | Cross-level examples |
|---|---|---|---|---|
| **Over-abstraction** | An interface with only one implementation | Delete the interface; use the implementation directly (unless the interface isolates a known volatile dependency for testability) | Every abstraction selects some details and omits others. With only one consumer, you don't know which details matter — you're locking in assumptions you can't yet validate. The abstraction will likely need to change when the second consumer arrives anyway. | Code: extra interface for a helper function. Module: repository interface when DB never changes. System: boundary layer between every layer. Process: a "process abstraction layer" (Jira → Trello board that mirrors Jira). API: a generic `/resource` endpoint that takes a type parameter. UI: a component abstraction for a one-time layout. |
| **Pattern-mapping** | "React has useState, so we should make one too" | Wait for a concrete use case first | Imports a foreign model's constraints without having the problem that model solves. The pattern carries assumptions from its origin context that may not hold. | Code: adding reactive state management to a CLI tool. System: adding microservices because "everyone does it" for a 3-person team. Process: implementing SAFe because "it's the standard" for a 5-person startup. API: adding GraphQL because "it's modern" for two static consumers. |
| **Dead code retention** | "This function might be useful later" | Delete it. Git history remembers it. | Code that isn't used still must be read, understood, and maintained. It obscures what's actually in use and creates false confidence ("there's a function for that" — but it's not tested). | Code: unused utility functions. Module: unused interface methods. System: dead controller endpoints. API: deprecated fields kept "for compatibility" with no migration plan. |
| **Config bloat** | Adding an env var to toggle a new feature, a configuration file for every module, or a "settings" page full of rarely-changed options | Code is configuration; prefer declarative API over YAML/JSON/env-var assembly. If you can express the variation in code, do it there. | Each config point is a decision deferred to runtime — which means it's a decision that can be wrong at runtime without being caught by the compiler/tests. Configuration is a liability, not a feature. | Code: env var for log level. System: a `config.yaml` with 50 settings where only 3 ever change. API: query parameters for every optional field. UI: a preferences page with 30 toggles. Process: a "process configuration" system that teams must customize before starting work. |
| **Doc padding** | Generating JSDoc for every function, documenting every parameter with "the parameter" | Comment only when semantics aren't obvious from code. Good design shrinks documentation. | Every maintained comment is a liability — it can rot, and it slows down reading. When code changes and the doc doesn't, the doc actively misleads. | Code: `/** @param {string} name The name */`. API: a 50-page OpenAPI spec nobody reads. UI: design system documentation that describes every prop. Process: a 20-page onboarding document for a tool that should be intuitive. |
| **Toolchain bloat** | Adding a new build step, plugin, transpiler, formatter, linter rule | Default: don't add. Ask: can existing tools do this? | Each toolchain addition is a dependency: it can break, become abandoned, require upgrades, and add time to every build. The cost compounds across the team. | Code: adding Babel plugin for a syntactic sugar used twice. System: adding a service mesh for 3 services. Process: adding a code review tool that does the same thing as the git platform. API: adding an API gateway for 2 endpoints. UI: adding a CSS framework for a 3-page app. |
| **Test-only public methods** | Changing private to public just for testing | Test through the public API, or rethink the design | Tests that couple to implementation details break when the implementation changes — even when behavior is preserved. The test becomes a liability instead of a safety net. | Code: exposing internal state for a unit test. Module: making a component's internal API public for integration tests. Systems: opening a database port to the test environment instead of testing through the business layer. |
| **Defensive future-proofing** | "Might need to support X later, so wrap it in an abstraction now" | Later's problem stays in the future | Most predicted futures never arrive. The abstraction you add today for a future that never materializes is pure cost — cognitive load, indirection, and the illusion of flexibility (an abstraction tested against zero real-world scenarios is fragile). | Code: generic `processData<T>` before any T variant exists. System: multi-cloud abstraction when running on one provider. Process: a "scale-ready" process for a 10-person team. API: versioned endpoints before the first consumer. UI: a responsive layout for a desktop-only internal tool. |
| **Premature generalization** | "Let me make this generic so it handles all cases" | Handle the one case you have. Generalize only when the second case arrives. | Without two concrete examples, you're guessing the commonality. Half the time the cases diverge in the opposite direction from your guess, and the "generic" code becomes a straitjacket you must refactor around. The cost is paid by every future reader who must understand the unused generality. | Code: a `sortByField(field, order)` utility before any call needs dynamic sorting. System: a plugin system before any plugin exists. API: a polymorphic endpoint accepting union types when only one type is ever sent. |
| **Silent complexity** | Hiding a significant decision behind a thin wrapper | Make complexity explicit in the caller's code, not buried in a helper | The wrapper gives the illusion of simplicity while the complexity still exists — it's just hidden. When the wrapper's assumptions break, debugging requires understanding both the wrapper and the hidden complexity. | Code: `cache(result)` wraps Redis cluster management. System: `@Transactional` hides a distributed transaction. API: an SDK that silently retries 5 times. UI: a `MagicButton` component that does async data loading. |
| **Speculative refactoring** | "While I'm here, let me clean up this unrelated code" | Stay in scope. Create a separate task. | Unrelated changes introduce risk without proportional benefit. They make code review harder (unrelated changes mixed together), and they can introduce regressions in areas that weren't supposed to change. | Code: renaming variables across unrelated modules during a bug fix. System: migrating a database while adding a feature. Process: reorganizing the team chart during a release. |
| **Framework envy** | "X framework does it this way, so our system should mirror it" | Choose idioms that match the language, runtime, and problem domain | Frameworks are opinionated about their domain. Importing those opinions into a different domain introduces constraints that don't serve your problem. The constraint looks harmless ("just a pattern") but grows into an impedance mismatch. | Code: adding React hooks pattern to a Go HTTP handler. API: RESTful HATEOAS for a CLI tool. UI: mobile-first patterns on a desktop dashboard. |
| **Silent error state** | Errors caught and recorded internally but upstream consumers ignore that state or override it with hardcoded `"completed"` / `"success"` / `True`. The system says "done" when everything actually failed. | Never hardcode a success status. Default to conservative — if you can't confirm success, assume failure. Trace error propagation end-to-end from catch site → state → caller → user output. | Creates a fake "all green" surface while the system silently does nothing useful. Every guard passes, every retry loop is a no-op. Hard to debug because observable output (status, logs) contradicts actual system state. | Code: try-catch that logs but returns `200 OK`. System: health check endpoint that pings itself instead of its dependencies. API: `{status: "ok"}` when the downstream call failed but was caught. Process: a CI pipeline that always passes because failing steps are configured as "optional." |

## Level-Specific Anti-Patterns

These anti-patterns are unique to specific Clean levels and don't have easy analogies across levels:

### Code
- **Primitive obsession:** Using string/int instead of a small value object. Correct: `new EmailAddress(s)` instead of `string s`.
- **Long parameter list:** >3 parameters without a good reason. Correct: bundle related params into an options object.
- **Shotgun surgery:** One change requires edits across many files. Correct: identify the missing abstraction and extract it.

### System
- **Service locator pattern:** Hiding dependencies in a global registry instead of injecting them. Makes dependencies invisible, runtime-fragile.
- **Anemic domain model:** Entities are data bags with no behavior. All logic in services. The "Clean Architecture with nothing in the center."
- **God component:** One component that everything depends on. Violates the Stable Dependencies Principle by making a stable component have many dependents.

### Process
- **Standup as status report:** 15 minutes of "what I did yesterday." Correct: focus on blockers and coordination.
- **Estimation-as-commitment:** Treating estimates as promises. Correct: estimates are ranges with confidence levels.
- **Retro-as-complaint-session:** Listing everything wrong without action items. Correct: each negative must generate one actionable improvement.

### Practitioner
- **Hero culture:** Staying late to "save the project." Correct: sustainable pace, honest communication about overcommitment.
- **The coward's "yes":** Agreeing to an impossible deadline because saying no is uncomfortable. Correct: "I can't commit to that scope by that date. Here's what I can do."
- **QA as someone else's problem:** Shipping without testing because "QA will catch it." Correct: own your quality.

### API
- **Leaky implementation:** Returning internal IDs, database columns, or stack traces in the response. Correct: the API contract is a boundary — hide internals.
- **Over-fetching / under-fetching:** Returning too much data (all fields) or too little (requiring N+1 calls). Correct: design responses for the consumer's use case.

### UI
- **Over-designed:** Multiple font sizes, 6+ colors, shadows on everything, gradients "for depth." Correct: 2 fonts, limited palette, consistent spacing.
- **Empty states ignored:** A component that renders "nothing" when there's no data (blank page, broken layout). Correct: every component must handle loading, empty, error, and success states.
